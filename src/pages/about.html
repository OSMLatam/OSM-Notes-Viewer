About OSM Notes Viewer

What is this?

OSM Notes Viewer is an interactive web application that visualizes analytics and statistics about OpenStreetMap notes activity around the world.

It provides insights into user contributions, country-level statistics, hashtag trends, and temporal patterns of note creation and resolution.

Manifesto: Our Perspective on OSM Notes

The Importance of Notes in OpenStreetMap

OpenStreetMap Notes are a fundamental mechanism for community collaboration and quality improvement. They serve as a bridge between mappers and the real world, allowing anyone to report issues, suggest improvements, or request new features directly on the map.

Notes represent the collective effort of thousands of contributors working together to improve the map's accuracy and completeness. Each note tells a story‚Äîa missing road, an incorrect name, a new business, or a mapping error that needs correction.

Notes help keep the map alive and up-to-date. They enable users who don't know how to map to report discrepancies between what they see on the street and what appears on the map. Mappers then take these reports to provide feedback to the map, improving it according to what users describe. This creates a continuous feedback loop that ensures the map reflects the real world as accurately as possible.

Our Mission

Our mission is to make OSM Notes more accessible, understandable, and actionable. We believe that by providing better tools and insights, we can:

  • Help mappers identify and resolve notes more efficiently
  • Encourage new contributors to engage with the notes system
  • Provide transparency into the notes ecosystem and community activity
  • Support research and analysis of mapping patterns and trends
  • Foster collaboration within the OpenStreetMap community

The Latin American Community

There is a dedicated community in Latin America focused on resolving OSM Notes. This community works tirelessly to improve the map quality across the region, addressing notes related to:

  • Missing or incorrect road networks
  • Place names and administrative boundaries
  • Points of interest (POIs) and businesses
  • Geographic features and natural landmarks
  • Data quality improvements

This project is part of that effort, providing tools and insights to support the community's work.

Rustiruz coined the term "notathon" as an activity where people gather to resolve notes. Since then, notathons have been held in several Latin American countries, and people who resolve notes have been brought together in a Telegram channel where they support each other.

These notathons have allowed the Latam community to reconsolidate, reaching the point of resuming State of the Map Latam events.

Returning to notes, the lack of tools for note resolution has led to requests for many applications to make changes for notes, from including the app name and version, to including specific functionality (fediphoto, damn). However, in terms of resolution, the tools are limited: they show the location of notes (notesViewer), the number of open or closed notes (HDYC), but nothing more. Analysis is scarce, like the notes heatmap app.

On the other hand, notes were added after implementing OSM API 0.6, and their functionality is very limited. Querying a user's notes is tedious, and those of a country are only possible with Pascal Neis's resultMaps, but its analysis is limited.

Additionally, there was a desire to recognize the great notes resolvers of Latam, and that's how the use of the planet dump began. Gradually, the idea matured of having a notes contribution profile similar to HDYC, and that's when research began on API queries for updated data.

That was the Eureka moment, and a mechanism was imagined to download the history and then keep it updated. The first part was in bash, which downloaded the history. Progress continued, but there was a blocker: processing the Planet XML is challenging, and several tools crashed. Years later, his friend Jos√© Luis showed him Cursor IDE and how to develop things, where he began practicing with GeoGame, a project that will be resumed once this viewer is stable.

Thanks to AI and many prompts, the proposal was made not to use XML tools, but rather AWK, a robust command with many years of experience. That's how it was possible to process all notes without problems and move forward with the project in late 2025.

Thanks to AI, the code has become more robust, correcting hundreds of errors, including unit tests, integration tests, GitHub Actions, dividing the code into multiple repositories, improving SQL, separating the ingestion part from the analytics part, as well as factoring the common part into commons. AI in 2025 accelerated the process and has been a learning experience, as AI sometimes hallucinates, and that's why it must be given good context to focus it. That's why all these repos are full of documentation, so that any future improvement is documented, and any AI can improve the code.

What We've Built

This project is part of a comprehensive ecosystem consisting of 8 repositories and 2 databases working together to process, analyze, and visualize OSM Notes. The ecosystem includes data ingestion pipelines, analytics systems, REST APIs, web map services, monitoring tools, and visualization interfaces. All components are open source and available for the community to use, modify, and extend.

For detailed information about each component, see the Ecosystem Architecture section below.

Ecosystem Architecture

This viewer is part of a comprehensive ecosystem of 8 repositories and 2 databases working together to process, analyze, and visualize OSM Notes:

Repositories

  1. OSM-Notes-Ingestion: Downloads and maintains OSM notes data from Planet dumps and API. Handles incremental updates and data synchronization.
  2. OSM-Notes-Common: Shared libraries and utilities used across all projects. Includes validation functions, error handling, and common Bash scripts.
  3. OSM-Notes-Analytics: Data warehouse and ETL system that processes the ingested data and generates statistics. Creates datamarts and exports JSON/CSV files. Includes ML-powered recommendations for note actions (close, comment, map).
  4. OSM-Notes-Viewer: This web application that visualizes the analytics data. Provides user profiles, country statistics, and interactive visualizations.
  5. OSM-Notes-Data: Data repository serving pre-generated JSON files via GitHub Pages/CDN. Contains user profiles, country statistics, and indexes.
  6. OSM-Notes-API: REST API providing dynamic data access. Handles individual note details, hashtag statistics, ML recommendations, and note search.
  7. OSM-Notes-Monitoring: Monitoring and observability system. Provides Grafana dashboards, Prometheus metrics, and system health monitoring for the entire ecosystem.
  8. OSM-Notes-WMS: GeoServer configuration and WMS (Web Map Service) setup. Contains GeoServer workspace configuration, layer definitions, and styling for map visualization services.

Databases

  1. PostgreSQL (Ingestion Database): Stores raw notes data from OSM Planet dumps and API. Contains tables for notes, note_comments, users, and countries. Used by OSM-Notes-Ingestion for data storage and incremental updates.
  2. PostgreSQL (Analytics Data Warehouse): Data warehouse with star schema design. Contains fact and dimension tables for analytics processing. Used by OSM-Notes-Analytics for ETL processes and generating statistics. Includes datamarts for user profiles, country statistics, and temporal analysis.

Services

  • GeoServer (WMS): Web Map Service providing map layers for visualization. Configured via OSM-Notes-WMS repository. Serves layers for open notes, closed notes, and country boundaries.
  • ML Recommendations: Integrated within OSM-Notes-Analytics, provides machine learning-powered recommendations for note actions (close, comment, map).

Data Flow

Data flows through the ecosystem as follows:

  1. OSM Planet/API ‚Üí Raw notes data
  2. OSM-Notes-Ingestion ‚Üí Downloads and stores in PostgreSQL (Ingestion DB)
  3. OSM-Notes-Analytics ‚Üí Processes data into Data Warehouse (Analytics DB)
  4. OSM-Notes-Analytics ‚Üí Exports JSON files to OSM-Notes-Data repository
  5. OSM-Notes-Data ‚Üí Serves static JSON files via CDN
  6. OSM-Notes-Viewer ‚Üí Fetches and displays data
  7. OSM-Notes-API ‚Üí Provides dynamic REST API access
  8. OSM-Notes-WMS (GeoServer) ‚Üí Provides WMS map layers for visualization
  9. OSM-Notes-Analytics (ML) ‚Üí Provides recommendations via API
  10. OSM-Notes-Monitoring ‚Üí Monitors system health and performance

The analytics backend extracts notes data, transforms and aggregates statistics, and generates pre-calculated JSON files every 15 minutes.

Data Origins

All data displayed in this viewer originates from OpenStreetMap (OSM), including both cartographic data and notes data. The cartographic information is used for geographic visualization and mapping, while the notes data provides the statistics and analytics shown throughout the application.

To improve the accuracy of maritime zones representation, data from Marine Regions was used as a reference. This allowed us to enhance the maritime boundaries and zones when attempting to replicate the information from that portal using OSM data, ensuring better alignment with recognized maritime boundaries and Exclusive Economic Zones (EEZs).

Licenses: OpenStreetMap data is licensed under the Open Database License (ODbL). Marine Regions data is licensed under CC-BY 4.0.

Technology

This is a fully static web application built with vanilla HTML, CSS, and JavaScript. It requires no backend server and fetches all data from pre-generated JSON files.

Stack:

  • HTML5 & CSS3
  • Vanilla JavaScript (ES6 modules)
  • Chart.js for visualizations
  • Deployed on CDN for global access
